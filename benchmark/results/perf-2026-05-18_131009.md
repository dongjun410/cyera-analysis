## Methodology

- **Measurement:** Per-document end-to-end latency (includes L1+L2 inference for all models)
- **Warmup:** 3 documents excluded from measurements
- **Batch size:** 1 (single-document inference)
- **Scoring:** Median latency as primary metric (robust to outliers); avg, min, max, P95 also reported
- **GPU mode:** Gemma via Ollama default GPU; FLAN-T5 via `model.to("cuda")`
- **CPU mode:** Gemma via `NUM_GPU=0` in service (num_gpu=0 in Ollama options); FLAN-T5 via `device="cpu"`
- **Note:** Current gemma-doc-label service has exact-match label validation that causes L1 retries (up to 3×). This inflates Gemma latency ~3× vs. single-pass inference.

## 6. Known Issues

### 6.1 Gemma service label matching inflates latency
The gemma-doc-label service uses exact snake_case matching for label validation. When Gemma returns abbreviated labels (e.g. `"legal"` instead of `"legal_and_compliance"`), the service retries up to 3× for L1. Since L1 never matches, L2 is never called. This means:
- Each document triggers up to 3 L1 Ollama calls (instead of 1 L1 + 1 L2)
- CPU mode particularly affected (3× ~2.3s = ~7s vs expected 2× ~2.3s = ~4.6s)
- Fixing label matching would reduce Gemma latency ~2-3×

### 6.2 Dspm27 PDF extraction overhead
Dspm27 uses pdfplumber to extract text from PDFs. The "Cannot set gray non-stroke color" warnings are harmless pdfplumber rendering warnings. Extraction time is included in per-doc latency.

### 6.3 FlanT5 pipeline compatibility
FlanT5ClassificationModel uses direct `model.generate()` call instead of HuggingFace pipeline, because recent transformers versions (5.8.1) removed `text2text-generation` task support for T5 models in the pipeline API.
