## 2. flan-t5-classification(large) Investigation

### 2.1 Root Causes Found

| # | Severity | Issue | Location |
|---|:-------:|-------|----------|
| 1 | **High** | `max_length=512` tokenizer truncation starves document content when datasets have many L1 categories | `flan_t5_classification.py:73` |
| 2 | **High** | `_fuzzy_match` cannot match snake_case outputs (e.g., `financial_reports` → no match to `Financial Reports`) | `flan_t5_classification.py:195-213` |
| 3 | **Low** | `.split("", 1)` typo in `_predict_single_step` (empty string separator, accidental no-op) | `flan_t5_classification.py:184,186` |
| 4 | **Design** | No output format constraint — model can output `"Category: Financial Reports"` instead of just `"Financial Reports"` | `flan_t5_classification.py:6-28` |

### 2.2 Evidence for Issue #1 (512-token truncation)

Token budget analysis for the L1 prompt (template + label options + document text):

| Dataset | L1 Categories | Prompt Overhead | Remaining for Text | L1 Accuracy |
|---------|:------------:|:---------------:|:------------------:|:-----------:|
| Cxh5types | 3 | ~35 tokens | ~475 tokens | 79.8% |
| Ben25 | 9 | ~60 tokens | ~450 tokens | 64.0% |
| Dspm27 | 14 | ~80 tokens | ~430 tokens | **51.9% (same as base)** |

The fact that base and large models produce identical L1 on Dspm27 is the smoking gun — neither model sees enough document content to differentiate; the bottleneck is input truncation, not model capacity.

### 2.3 Evidence for Issue #2 (weak fuzzy matching)

Comparison of FLAN-T5 vs Gemma fuzzy matchers on common model output patterns:

```
Input: 'financial_reports'     FLAN [FAIL] -> 'financial_reports'  (raw pass-through)
                              Gemma [OK]   -> 'Financial Reports'

Input: 'HR_Records'            FLAN [FAIL] -> 'HR_Records'
                              Gemma [OK]   -> 'HR Records'

Input: 'financial_report'      FLAN [FAIL] -> 'financial_report'
                              Gemma [OK]   -> 'Financial Reports'
```

FLAN-T5, as a text-to-text model, frequently outputs labels in training-data formats (snake_case, underscores). The old matcher only checked case-insensitive exact and substring matches — neither handles underscore-to-space conversion.

### 2.4 Fixes Applied

**Fix 1** — Increase context window (`flan_t5_classification.py:73`):
```diff
-            max_length=512,
+            max_length=1024,
```

**Fix 2** — Upgrade fuzzy matcher to snake_case-aware matching (`flan_t5_classification.py:195-234`):
- Added `_to_snake()` static method (normalize `&`, parens, non-alphanumeric → underscores)
- Exact match: case-insensitive + snake_case identity
- Substring match: snake-normalized with min 3-char threshold
- Word-level overlap: fallback for partial matches
- Returns `"unknown"` on no match instead of raw unmatched string

**Fix 3** — Fix `.split("", 1)` typo (`flan_t5_classification.py:184,186`):
```diff
-            l1_pred = line.split(":", 1)[-1].split("", 1)[-1].strip()
+            l1_pred = line.split(":", 1)[-1].strip()
```

**Fix 4** — Add output format constraints to all three prompts (`flan_t5_classification.py:6-28`):
- L1 prompt: added `"Output ONLY the category name, nothing else."`
- L2 prompt: added `"Output ONLY the subcategory name, nothing else."`
- Single-step prompt: changed to `"Output ONLY the result below, nothing else:"`

### 2.5 Effect of Fixes

```

Dataset         Original   +trunc+fuzzy   +format constraint     Gemma4
-------------------------------------------------------------------------
ben25             0.6400         0.6800               0.6800     0.8400
```

## 3. doc-classifier-sklearn Investigation

### 3.1 Root Cause: Label Taxonomy Mismatch

The original `DocClassifierSklearnModel` wrapped a pre-trained model from `ZerosOne/doc-classifier/` whose code path (`dspm_best`) revealed it was trained exclusively on Dspm data.

**Critical bug**: `predict_labels()` completely ignored its `l1_options` and `l2_options` parameters:
```python
def predict_labels(self, texts, l1_options, l2_options):
    results = []
    for text in texts:
        result = self._classify_one(text)  # uses internal fixed label set
        results.append(result)
    return results
```

Contrast with Gemma/FlanT5 which use the provided label options to construct prompts and map outputs. The sklearn model was answering a **different classification question** than what the benchmark was measuring:

| Dataset | Model's Label Taxonomy | Benchmark's Label Taxonomy | Match |
|---------|----------------------|--------------------------|:-----:|
| Dspm27 | Dspm labels | Dspm labels (same source) | 85.2% |
| Ben25 | Dspm labels | Ben25 labels (different taxonomy) | 16.0% |
| Cxh5types | Dspm labels | Cxh5types labels (different taxonomy) | 25.2% |

The 16.0% and 25.2% were not model capability issues — they were **label space mismatches** where the model's output label names simply didn't correspond to what the benchmark was measuring against (exact string comparison).

### 3.2 Fix: Per-Dataset Native Training (Plan B)

**Why Plan B (not label mapping):** Unlike Gemma/FlanT5 which are zero-shot (labels injected at inference time), sklearn's LogisticRegression has a fixed classification head trained on specific class labels. A label mapping could fix format differences but not fundamentally different taxonomies.

**Implementation** (`doc_classifier_sklearn.py` — complete rewrite):
- Added `fit(texts, labels)` method: trains TF-IDF vectorizer (max_features=8000, 1-2 grams) + LogisticRegression (C=1.0, lbfgs) on L1, plus per-L1 L2 classifiers
- `predict_labels()` uses trained native model when fitted, external model as fallback
- Removed `multi_class` parameter for sklearn 1.8 compatibility

**Benchmark script modifications** (`test_performance.py`, `compare_accuracy.py`):
- Added 80/20 train/test split with stratified sampling (fallback to random for rare classes)
- All models now evaluated on the **same hold-out test set** for fair comparison
- `compare_accuracy.py`: accuracy-only comparison script

## 5. Key Conclusions

### 5.1 Code issues, not model issues, caused the original anomalous results

- **flan-t5-classification(large)**: The 512-token truncation and weak fuzzy matching suppressed accuracy by 3-4pp. But even after fixes, the 780M encoder-decoder model cannot close the gap to Gemma4 (2B decoder-only) — the remaining 11-16pp gap is model capacity.
- **doc-classifier-sklearn**: The 16-25% figures were entirely due to cross-taxonomy label mismatch (Dspm model applied to non-Dspm labels). When trained on the correct taxonomy, sklearn achieves 100% L1 on Cxh5types.

### 5.2 Architectural trade-offs verified

| Dimension | sklearn (TF-IDF+LR) | FlanT5-large | Gemma4:e2b |
|-----------|:------------------:|:------------:|:----------:|
| Paradigm | Per-dataset training | Zero-shot | Zero-shot |
| Data requirement | Needs labeled training data | None | None |
| L1 accuracy* | 100.0% (when trained) | 78.8% | 96.2% |
| Latency | 0.001s | 0.37s | 1.10s |
| Speed vs Gemma | 1000x faster | 3x faster | baseline |
| Generalization | Same-distribution only | Moderate | Best |
| Production readiness | Needs training data per taxonomy | Ready, moderate accuracy | Ready, best accuracy |

*Cxh5types 52-doc test set

### 5.3 Output format constraints had no effect on FLAN-T5

Adding `"Output ONLY the category name, nothing else."` to prompts did not improve accuracy. FLAN-T5-large's instruction-following capacity is insufficient to benefit from format constraints. This is a model limitation, not a prompt engineering gap.

### 5.4 Cxh5types is synthetic template data — benchmark utility limited

Cxh5types documents are highly templated (JSON invoices, payslip templates, standard NDA forms). The 100% L1 achieved by sklearn reflects that each L1 class has near-orthogonal vocabulary patterns — a single keyword like `"PAYSLIP"` or `"NON-DISCLOSURE"` perfectly discriminates. This makes Cxh5types suitable for:
- **Sanity checking** (any model should perform well here)
- **Pipeline validation** (does the benchmark framework work end-to-end)

It is **not** suitable for estimating production generalization.

### 5.5 fair comparison requires shared test split

The original benchmark compared sklearn (trained, evaluated on test split) against Gemma/FlanT5 (zero-shot, evaluated on full dataset). Methodological fix: all models must be evaluated on the same hold-out test set, with sklearn trained only on the training partition.

### 5.6 Small datasets limit sklearn but not zero-shot models

Ben25 (25 docs) and Dspm27 (27 docs) are too small to train sklearn reliably (14 L1 categories from 21 training examples = most classes have 1-2 samples). Zero-shot models (Gemma, FlanT5) are unaffected by training data scarcity.
