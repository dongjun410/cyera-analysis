# DataDNA优化方案 vs V2.2 — 终版全面对比与评价

> 对比日期：2026-05-22
> 修订日期：2026-05-22 (v2.0 — 中立审查后修订)
> 对比对象：
> - 方案A: 企业文档智能聚类系统V2.2_实施方案_含知识蒸馏 (下称 V2.2)
> - 方案B: DataDNA分类引擎—工程最优实现方案 v1.1 (下称 本方案)

---

## 零、对比方法论

以下每条差异按三个维度评价：

- **技术正确性**：从第一性原理出发，哪个方案在该维度上更符合问题本质
- **工程成熟度**：哪个方案更接近可部署的生产系统
- **适用场景**：各自在什么条件下是更优选择

本对比努力保持中立，但完全客观是不可能的——两份文档处于软件开发生命周期的不同阶段（V2.2是可执行代码，本方案是架构设计），直接对比需注意语境的差异。

## 一、架构哲学：一体化管道 vs 分层递进

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| 架构模式 | 线性 Phase 0→1→2→3→4→5→6 | 分层 Tier 0→1→2→3 + 离群循环 |
| 组件边界 | 模块间共享数据结构，紧密耦合 | 每层独立输入/输出，松耦合 |
| 错误传播 | 单点故障影响全链路 | 每层有独立降级路径 |
| 复杂度来源 | 模块数量多(10个核心模块)，但每个相对简单 | 模块少但每层逻辑更复杂 |

### 评价

**V2.2 的优势在于可执行性。** 每一个模块都有完整的 Python 代码实现，有明确的文件结构、config.yaml、CLI入口。一个工程师拿到文档可以直接开始写代码。这是"从零到一"的正确姿势。

**本方案的优势在于可扩展性。** 分层架构使每层可以独立演进——换掉 DeBERTa 不影响 Tier 1，升级 Mistral 版本不影响 Tier 0。松耦合在系统需要运行 3 年以上时变得至关重要。但分层架构的代价是每个 Tier 的接口设计和数据契约需要更仔细的前期设计，也会引入新的 bug 面和调试复杂度。

**第一性原理判断：** 数据安全分类的真实世界需求是多样化的——有些客户只需要 PII 检测，有些需要全文档分类，有些需要类型发现。分层架构允许按需启用/禁用 Tier，比一体化管道更灵活。但分层架构在简单场景下可能是过度设计（over-engineering）——如果客户的文档类型高度同质，一体化管道完全够用。

**结论：** 两种架构各有适用场景，不是简单的"谁替代谁"。V2.2 适合快速启动和文档同质场景。本方案适合需要长期演进和文档异构场景。一个合理的工程路径是先用 V2.2 跑通端到端流程，然后按瓶颈驱动将各模块演进到本方案的分层架构。但这不是唯一路径——如果架构升级的收益无法用基准测试量化证明，保持在 V2.2 的架构上做增量改进也可能是更务实的选择。

## 二、聚类方法论：KMeans vs 结构哈希+语义精炼

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| 聚类核心 | 全局 KMeans (基于联合嵌入) | Stage A: 结构哈希, Stage B: FAISS语义精炼 |
| 嵌入使用 | 语义嵌入 + 结构特征向量拼接 → 联合聚类 | 语义嵌入仅用于Stage B簇内精炼 |
| K值 | Silhouette Score网格搜索 | 自动（结构种类 × 语义分层） |
| 模板化文档 | 与内容型文档混合聚类 | 结构阶段先分离，各自精炼 |
| 增量友好性 | kNN最近邻 + 缓冲区 | 哈希定位(O(1)) + FAISS检索(O(log N)) |
| 可解释性 | 低（嵌入空间黑箱） | 高（可追踪到结构特征） |

### 评价

**这是两个方案之间工程影响最大的差异。**

V2.2 的 KMeans 方案将**结构相似性**和**语义相似性**压缩到同一个向量空间中，通过加权拼接来实现。这在概念上优雅（"一个向量表示一切"），但在实践中存在权衡：

- 结构特征向量（19维）和语义嵌入（1024维）在拼接后，KMeans 的欧氏距离被高维语义嵌入主导。给结构特征加权重 `structure_weight=0.3` 是一种启发式补偿，但这个权重的正确值取决于文档集合——没有普适的最优值。
- 模板化文档（如1000份格式相同、金额不同的发票）和内容型文档（如1000篇主题各异的政策备忘录）在同一个嵌入空间中被聚类。前者的理想簇划分由结构驱动，后者由语义驱动。KMeans 无法同时优化两个不同的目标。

本方案的两阶段聚类从结构上解决了这个问题：结构阶段按"文档长什么样"分桶，语义阶段在桶内按"文档说什么"做细粒度分组。两个独立的问题，两个独立的解决方案。

**但这不是没有代价的：** 两阶段聚类增加了系统复杂度（双阶段调参、Stage B触发条件、增量分配中的桶内离群管理）。此外，本方案的两阶段聚类完全未实现，所有关于FAISS IVF + HDBSCAN的可扩展性声明（"O(N log N) 而非 O(N²)"）都是理论分析，没有实际基准测试验证。

**关于 KMeans K 值搜索的成本：** V2.2 的 `_find_optimal_k` 在候选 K=10 到 200 步长 5 上做 KMeans，加上精搜，大约需要跑 40+ 次 KMeans。对于 10 万文档 × 1043 维（1024 语义 + 19 结构），单次 KMeans 约 2-5 秒（GPU 加速），总搜索时间在 2-4 分钟。这个开销对于全量批处理是可接受的，但对于需要频繁重聚类的场景会成为瓶颈。

**第一性原理判断：** 聚类是一个**双目标优化问题**——你既要让结构相似的文档在一起，又要在结构相似的文档中区分语义差异。将两个目标压缩到单一向量空间是对问题的过度简化，两阶段聚类是更诚实的解决方案。但"更诚实"不自动等于"在实践中有可测量的收益"——这个收益需要通过对比实验（同一批文档上两种聚类方法的纯度/完整性对比）来验证。

**结论：** 本方案的聚类方法论在概念上更正确，但未经验证。V2.2 的方案已可实现且对文档同质场景可能完全够用。如果你的文档类型高度同质（比如全是合同，或全是邮件），V2.2 的单一聚类就够了。如果你的数据源多样（PDF/DOCX/XLSX/JSON/EML 混合），本方案的两阶段聚类值得尝试，但应先在小规模数据上对比验证收益再决定是否引入。

## 三、NER/PII检测：Presidio vs 分层规则+DeBERTa

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| PII 检测 | Presidio (Microsoft开源) | Tier 0 规则引擎 (基于Presidio规则+自扩展) |
| 上下文消歧 | 无 | DeBERTa-v3 上下文判别 |
| 误报控制 | 置信度阈值 (Presidio内置) | 规则层上下文标记 → DeBERTa消歧 → LLM仲裁 |
| PII 的角色 | **分类决策者** (高PII密度→预分类标签) | **特征提供者** (PII特征注入Tier 2，不直接决定文档类别) |
| 噪声鲁棒性 | 弱 (代码/测试数据/日志中误报高) | 强 (三层过滤) |

### 评价

V2.2 的 PII 预分类逻辑存在一个值得关注的风险点：**高PII密度的文档被直接赋予分类标签，绕过了聚类流程。**

```python
# V2.2 pii_preclassifier.py 的核心逻辑
PRECLASSIFICATION_RULES = {
    "PII-heavy: Contact/Directory": {PIIType.PHONE: 5, PIIType.EMAIL: 5},
    "PII-heavy: Financial/Payment": {PIIType.CREDIT_CARD: 2},
    "PII-heavy: Identity Documents": {PIIType.SSN: 2},
    "PII-heavy: Banking Records": {PIIType.IBAN: 3},
}
```

这意味着如果一篇文档中检测到 2 个以上信用卡号，它就被直接分类为"金融/支付类文档"。以下场景会导致误报：
- PCI-DSS 合规培训文档（含示例信用卡号）→ 被标记为"金融/支付"
- 代码仓库中的单元测试文件（含测试 SSN）→ 被标记为"身份文件"
- 讨论信用卡欺诈的新闻简报 → 被标记为"金融/支付"

**但这个问题的严重性需要结合上下文评估：**
1. V2.2 的 `confidence_threshold=0.8` 意味着低于置信度阈值的PII检测不参与计数，只有高置信度匹配才触发预分类。
2. `PRECLASSIFICATION_RULES` 是硬编码字典，在实际部署中可由用户按场景增删——如果客户环境包含大量代码文件，可以直接禁用 `SSN` 或 `CREDIT_CARD` 的预分类规则。
3. 预分类的定位是"快速路径优化"——它假设高PII密度与敏感文档类别高度相关，这在纯业务文档库（不含代码/测试数据）中是合理的近似。

因此，将这个问题定性为"深刻的架构问题"可能过度。更准确的描述是：**V2.2 的 PII 预分类是一个在特定假设下有效的优化，但在数据源多样化时存在误报风险。** 本方案将 PII 降级为特征之一，在方法论上更稳健，但代价是必须100%走聚类流程（放弃了 V2.2 的快速路径）。

**关于 DeBERTa 延迟优势的声明：** 本方案声称 DeBERTa 做 NER 的延迟是 FLAN-T5 生成式 NER 的 1/50 ~ 1/100。这个数字缺乏基准测试支撑。FLAN-T5 的 decoder 逐 token 生成确实比 encoder-only 慢，但实际差距取决于序列长度和批处理策略——短序列（<100 token）场景下差距可能只有 10-20 倍。建议在实际数据上做 benchmark 验证这个关键假设。

**第一性原理判断：** PII 的存在是分类的重要信号，但不是唯一信号。本方案将 PII 作为特征而非决策，在方法论上更稳健。但 V2.2 的预分类快速路径可以通过简单改进（增加上下文惩罚词检测、增加预分类标签的人工审核队列）来降低误报风险，而不需要完全重新设计架构。

**结论：** 本方案的 PII 策略在方法论上更稳健，尤其适合数据源多样化的场景。V2.2 的简化策略在纯业务文档库中可工作，但应在部署前评估误报风险，并考虑增加上下文消歧机制。

## 四、LLM的角色与调用模式

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| 调用场景 1 | 簇命名 (Phase 4): 每个簇调用1次 | Tier 2 Step 4: 仅未匹配已知类型的簇 |
| 调用场景 2 | 蒸馏标注验证 (Phase 5.5): 每簇10篇文档 | Tier 3: 仅触发条件的高风险/低置信度文档 |
| 调用场景 3 | 增量fallback (incremental.py): 低置信度文档 | 离群文档描述 (类型发现循环) |
| 估计调用比例 | 15-25% 文档涉及 LLM 调用 | 5-10% 文档涉及 LLM 调用 |
| LLM 选型 | Qwen2.5-7B (统一) | Mistral-7B (4-bit/FP16按场景分层) |

### 评价

V2.2 的 LLM 使用存在一个方法论层面的考量：**同一 LLM（Qwen）既用于标注标签，又用于验证标签。**

1. Phase 4: LLM 为簇生成标签（`_llm_name_cluster`）
2. 模块10: LLM 验证文档是否匹配这个标签（`_llm_validate_label`）

如果 LLM 存在系统性偏差（例如倾向于将含医学术语的文档标记为"医疗记录"而不区分"医疗研究论文"和"患者病历"），第二次调用可能强化这个偏差。这是"让学生批改自己的试卷"——验证者和被验证者共享相同的错误模式。

**但这个问题的实际影响取决于偏差的严重程度。** 如果 LLM 在文档分类任务上的准确率本身较高（>90%），自我验证带来的额外误差可能很小。反之，如果 LLM 在特定领域的分类错误率较高，自我验证会放大问题。

本方案中，Mistral 验证层的验证对象来自不同来源（规则引擎、DeBERTa NER、结构签名匹配）。这三个来源的误报模式不同，LLM 作为第三方仲裁确实有真实的信息增益。但本方案也用了同一个 Mistral-7B 模型（仅量化配置不同），并非完全独立的验证者。

**缓解方案：** 无论采用哪个方案，都可以通过低成本改进缓解自我验证问题：
- 蒸馏验证时使用**不同的 LLM**（如 Qwen 标注、Mistral 验证）
- 使用**不同的 prompt 策略**（如要求 LLM 给出反面论据）
- 引入**人工抽检**作为最终的独立验证源

**关于 LLM 分类准确率的共同盲点：** 两份文档都假设 LLM（Mistral/Qwen）能正确理解企业文档的业务语义。但实际上，企业文档（合同、财报、合规文件、技术规范）的分类需要领域专业知识，通用 LLM 的错误率可能显著高于预期。没有一份文档讨论"如果 LLM 分类准确率只有 70% 怎么办"。

**结论：** 本方案的 LLM 验证设计在方法论上更严谨，但 V2.2 的自我验证问题可以通过低成本改进缓解。两者共享一个更根本的风险——LLM 在企业文档分类上的真实准确率未经基准测试验证。

## 五、知识蒸馏与类型发现：概念分离

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| "自学习" 指什么 | LLM标注→训练SetFit→快速分类 | 离群文档→缓冲区→周期性重聚类→新类型注册 |
| 解决的问题 | 分类**速度** (蒸馏) | 分类**覆盖范围** (类型发现) |
| 能否发现新类型 | **不能** (SetFit分类头固定N类) | **能** (聚类自动产生新类型簇) |
| 模块命名 | "Learned Classification" (模块10) | 知识蒸馏 (Section 5) + 类型发现 (Section 4.5) |

### 评价

V2.2 模块 10 做的事：LLM 标注已有簇 → 用这些标注训练 SetFit → SetFit 更快地分类新文档。这纯粹是**知识蒸馏**（Knowledge Distillation）——将大模型（LLM）的分类能力压缩到小模型（SetFit）。压缩后的模型只能识别 LLM 见过的类型，无法发现新类型。

本方案将两件事分开：
- **知识蒸馏**（Section 5）：LLM → SetFit，加速已有类型的分类
- **类型发现**（Section 4.5）：离群缓冲区 → 周期性重聚类 → 新类型注册，扩展系统的类型覆盖

两者都需要，但服务于不同的目标。

**第一性原理判断：** 命名确实不是小问题——当系统设计阶段混淆了两个不同的概念，实现阶段可能缺失类型发现功能。V2.2 的模块 10 做的是蒸馏，不是 Cyera 定义的 Learned Classification（自动发现专有数据类型）。建议 V2.2：(a) 将模块 10 更名为知识蒸馏，(b) 新增类型发现模块（可从本方案的 Section 4.5 获取设计参考）。

**但需注意：** 本方案的类型发现设计（离群缓冲区 + 周期性重聚类）虽然概念完整，但也是纯设计、零实现。其实际效果取决于多个未验证假设：离群文档的聚类质量、新类型簇的内聚度、LLM 对新类型的命名准确率。

**结论：** 本方案在这个维度上的概念建模是正确的。V2.2 需要修正模块10的定位和命名。但两者在类型发现方面都没有经过实际数据验证。

## 六、结构特征的角色：拼接 vs 独立维度

### 差异

| | V2.2 | 本方案 |
|---|---|---|
| 结构特征的使用方式 | 拼接为嵌入向量的一部分，参与全局KMeans | 作为聚类第一阶段(Stage A)的独立维度 |
| 是否修改文本 | 不修改 (和本方案相同) | 不修改 |
| PII特征的角色 | 预分类依据 + 结构特征向量的输入 | Tier 0 产出，注入 Tier 2 作为分类特征 |

### 评价

V2.2 的双通道架构（语义通道 + 结构特征通道）是一个有效的想法——用结构特征辅助语义聚类。但实现方式（19维结构向量 × 0.3 权重拼接到 1024 维语义嵌入）使结构特征的贡献容易被高维语义嵌入主导。这个权重的选择本质上是启发式的——0.3 对于"全是表格的文件夹"可能太小，对于"全是纯文本的文件夹"可能太大。

本方案将结构特征提升到聚类架构的核心位置：Stage A 完全由结构特征驱动，语义嵌入仅在 Stage B 参与。这避免了"结构特征 vs 语义特征"的权重权衡问题。但引入了新的复杂度：Stage A 的结构哈希是确定性分桶，桶的粒度（结构特征的离散化方式）成为新的调参对象。

**判断：** 本方案对结构特征的使用在异构文档场景下更合理。V2.2 的拼接策略在文档结构同质的场景下工作良好，且实现更简单。两者都是有效的设计选择，优劣取决于实际数据特征——应在目标数据上对比验证。

## 七、工程完整度对比

下表反映了两份文档的**定位差异**，不是质量高低。

| 维度 | V2.2 | 本方案 | 说明 |
|------|:---:|:---:|------|
| 可执行代码 | 完整 Python 实现 (~3600行) | 架构设计文档，无代码 | V2.2 是实施方案，本方案是架构设计 |
| 配置文件 | 完整 config.yaml + docker-compose | 硬件需求表 | — |
| CLI 入口 | main.py + incremental.py + distill.py | 无 | — |
| 数据模型 | Pydantic dataclass + schemas.py | 概念描述 | — |
| 测试 | 测试目录定义 (含测试用例框架) | 评估体系设计 (无测试用例) | 两者都未提供完整的可执行测试 |
| 模型数量 | 2个 (BGE-M3 + Qwen2.5-7B) | 3个 (BGE-M3 + DeBERTa-v3 + Mistral-7B) | 模型增多 = 部署复杂度增加 |
| 敏感度自适应聚类 | **有** (V2.2 真正创新) | 无等价功能 | 按PII密度分层聚类，V2.2独有特性 |
| 可观测性 | 基础 (Silhouette/DBI/CHI) | 完整 (10项指标 + 告警阈值 + 漂移检测) | V2.2有聚类质量评估，但缺业务级可观测 |
| 容错降级 | 无 | 6条降级路径 + 重试策略 | 本方案显著领先 |
| 数据安全 | 全本地部署 (主动设计选择) | PII脱敏 + prompt注入防护 + 嵌入隔离 | V2.2的全本地策略本身就是安全方案；本方案的安全措施针对混合/SaaS部署 |
| 冷启动路径 | 无 (假设文档批量存在) | Phase -1 缓冲层 + 零样本临时标记 | 本方案领先 |
| 端到端评估 | 聚类质量评估 (无ground truth) | 评估数据集构建 + 多层级指标 + 回归测试 | 本方案设计更完整，但同样未实现 |
| 部署便捷性 | Docker Compose 一键启动 | 需手动部署3个模型 + FAISS + ES | V2.2 在部署便捷性上显著领先 |
| 多语言支持 | spaCy多语言分词 + 语言检测 | BGE-M3多语言嵌入 | 两者都支持多语言，V2.2的spaCy方案更细致 |

### 评价

**V2.2 是一份实施方案**——它的目标是让工程师可以开始写代码。它提供了完整的代码、配置、文件结构。它缺少可观测性、容错、安全设计（混合部署场景下）不是因为设计者不知道这些，而是对于"第一版实现"来说这些不是阻塞项。先跑通端到端，再加监控——这是合理的工程节奏。

**本方案是一份架构设计**——它的目标是让架构师在编码前验证设计的正确性。它提供了完整的分层架构、数据流、模型选型论证、容错策略。它没有可执行代码不是因为做不到，而是因为架构设计阶段的目标不是写代码。

**需要特别指出：** V2.2 的敏感度自适应聚类（`sensitivity_adaptive_scheduler.py`）是本方案中没有等价功能的真正创新——按 PII 密度将文档分入高/中/低敏感度层级，对每层使用不同的聚类粒度。这既降低了高敏感文档的误分类风险，又提升了低敏感文档的聚类效率。本方案值得考虑吸收这个设计。

## 八、两份文档的共同局限

在对比两份文档时，发现以下问题**两份文档都没有充分解决**：

### 8.1 LLM 分类准确率的基准缺失

两份文档都假设 LLM（Mistral/Qwen）能正确理解企业文档的业务语义并给出准确分类。但企业文档（合同、财报、合规文件、技术规范、医疗记录）的分类需要领域专业知识，通用 LLM 的错误率可能显著高于预期。没有一份文档提供 LLM 在企业文档分类任务上的基准测试数据，也没有讨论"如果 LLM 分类准确率只有 70% 怎么办"的降级策略。

### 8.2 人工审核闭环缺失

两份文档都提到了"标记为 manual_review"，但都没有设计人工审核结果如何反馈到系统中。人工修正的标签是极其宝贵的训练信号——应该用于重训练蒸馏模型、更新已知类型库、修正规则权重。缺少这个闭环意味着人工审核的价值被浪费了。

### 8.3 多租户隔离

两份文档似乎假设单客户部署。在多租户 SaaS 场景中，不同类型库、嵌入索引、蒸馏模型需要在租户间隔离，且客户的已知类型库不应被其他客户的数据污染。两份文档均未涉及多租户架构。

### 8.4 嵌入模型升级策略

BGE-M3 模型本身会发布新版本。当嵌入模型升级时，所有已存储的 FAISS 索引、簇中心、已知类型库的语义嵌入中心都需要重建。这是一个高成本的运维操作，两份文档均未讨论升级路径。

### 8.5 成本模型

两份文档都讨论了硬件需求和延迟估算，但都没有提供端到端的成本模型。对于企业采购决策者，"跑这个系统每月要花多少钱（GPU 租赁/电力/运维人力）"可能是比"Silhouette Score 多少"更重要的问题。

## 九、总体评价

### V2.2 的优势

1. **可执行性。** 10 个模块的完整 Python 实现 ≈ 3,600 行代码。一个工程师可以在 1-2 周内搭建完毕并看到运行结果。
2. **部署简单。** 2 个模型（BGE-M3 + Qwen），Docker Compose 一键启动，单机友好。
3. **渐进复杂度。** 核心逻辑简单——解析→嵌入→KMeans→LLM命名→存储。没有多层 Tier 之间复杂的数据契约和接口设计。
4. **敏感度自适应聚类**是 V2.2 的真正创新——按 PII 密度分层聚类，本方案没有等价功能，值得被吸收。
5. **多语言支持细致。** spaCy 分词 + 语言检测 + 多语言 PII 模式（Presidio），比本方案的纯 BGE-M3 嵌入方案在分词层面更精细。

### V2.2 需要改进的地方

1. **PII 预分类直接决策**在数据源多样化时有误报风险——需要增加上下文消歧或降级为特征输入。
2. **LLM 自我验证**——标注和验证用同一个 LLM，缺乏独立的验证信号。
3. **模块 10 的命名和定位**——做的是知识蒸馏，不是 Cyera 定义的 Learned Classification（类型发现）。
4. **缺少生产可运维性**——容错降级、冷启动、分类漂移检测、数据安全（混合部署场景下）。

### 本方案的优势

1. **架构正确性。** 分层递进（聚类→分类→验证）比一体化管道更符合数据安全分类问题的本质。
2. **方法论严谨性。** PII 作为特征而非标签、LLM 独立验证、类型发现与知识蒸馏分离——三个关键决策在方法论上正确。
3. **生产可运维性。** 可观测性、容错降级、冷启动、数据安全——这些是生产系统必需的，本方案提供了设计蓝图。
4. **聚类方法论。** 两阶段（结构+语义）聚类在异构文档场景下优于全局 KMeans。

### 本方案需要改进的地方

1. **零代码验证。** 所有性能数据均为估算，所有架构声明的实际收益未经基准测试验证。
2. **已知类型库冷启动假设过于乐观。** "匹配率从 0% 逐步上升到 70-90%"缺乏依据。
3. **DeBERTa vs FLAN-T5 的 50-100 倍延迟差距**缺乏 benchmark 支撑。
4. **知识蒸馏的灾难性遗忘缓解方案不充分。** full retrain + 回滚是粗糙的策略。
5. **模型数量更多（3 vs 2），部署和版本管理更复杂。**
6. **缺少敏感度自适应聚类的等价设计。**

### 可行的整合路径

以下路径是**一种**合理的选择，但并非唯一选择——实际决策应取决于团队能力、时间约束和数据特征：

```
Week 1-4:   用 V2.2 的实现跑通端到端流程
            → 验证"聚类→LLM分类→蒸馏"这条链路在目标数据上有效
            → 同时建立 LLM 分类准确率的基准测试

Week 5-6:   引入本方案中低成本高收益的方法论修正
            → PII 预分类增加上下文惩罚词检测（低成本改进）
            → 蒸馏验证使用不同 prompt 策略或不同 LLM
            → 模块10更名为知识蒸馏

Week 7-8:   引入本方案的生产加固（如需要）
            → 可观测性指标 + 日志追踪
            → LLM 降级路径 + 重试逻辑
            → 数据安全措施（如使用混合/SaaS 部署）

Week 9+:    按瓶颈驱动、有基准测试验证地引入架构升级
            → 如果聚类质量不够 → 在目标数据上对比 Stage A 结构哈希 vs 当前 KMeans
            → 如果 LLM 调用太贵 → 实现已知类型匹配层，测量实际节省的调用量
            → 如果新类型频发 → 实现离群缓冲区 + 类型发现循环
            → 如果 NER 误报太高 → 在目标数据上对比 DeBERTa vs Presidio 的误报率
            → 如果 PII 预分类误报不可接受 → 将 PII 降级为特征信号

关键原则: 每次架构升级前,先在目标数据上用基准测试量化当前方案的瓶颈,
         然后验证新方案确实改善了该瓶颈,再决定是否引入。
         不要基于架构论证直接采纳——用数据做决策。
```

**核心结论：V2.2 和本方案不是竞争关系，也不是简单的"v1.0 vs v2.0"线性演进关系。它们是同一问题的两种不同阶段的产出物——V2.2 是可工作的代码，本方案是更好的设计蓝图。两者结合的正确方式不是全盘采纳某一方，而是：用 V2.2 快速验证核心链路，用基准测试识别真实瓶颈，用本方案的设计洞察指导针对性改进，每一步改进都用数据证明收益。**
