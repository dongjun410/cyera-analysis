# Cyera专利侵权风险分析

> 分析对象: `2026-05-22-datadna-optimized-design.md` (v1.1)
> 分析日期: 2026-05-22
> **声明: 本分析仅为技术层面的独立评估，不构成法律意见。专利侵权判定需要由专业专利律师基于具体司法管辖区的法律和完整权利要求书进行。以下分析基于公开的专利摘要和说明书内容，而非完整的授权权利要求书（claims）。**

---

## 一、分析方法

对Cyera的8项专利/申请逐一进行**要素对比分析**（element-by-element comparison）：

1. 提取每项专利的独立权利要求（independent claims）的核心要素
2. 判断本设计方案是否实施了**每一个**要素
3. 如果本方案未实施某一要素，则不构成字面侵权（literal infringement）

专利侵权需要被控方案覆盖权利要求的**全部要素**。缺少任一要素 = 不侵权。

## 二、逐专利分析

### 专利一~三: 数据发现家族 (US12026123, US12499083, US12566567)

**权利要求的核心要素:**
- 扫描云磁盘快照（snapshot scanning）
- 基于文件元数据识别数据存储
- 创建特定数据库格式的引擎
- 两阶段/部分读取的扫描策略
- 惰性挂载（lazy mount）
- 条件性克隆

**本方案对应内容:** 无。本方案是**分类引擎**设计，不涉及数据发现（扫描磁盘、挂载快照、创建数据库引擎等）。

**结论: 不侵权。** 两个方案解决不同的问题，无要素重叠。

---

### 专利四: US 12,299,167 B2 — 数据分类（已授权）

**权利要求的核心要素:**

| 要素 | 专利描述 |
|------|---------|
| 路径A | 对**数值型数据**应用启发式真值表，输出第一评分 |
| 路径B | 对**字符串数据**应用ML分类器，输出第二评分 |
| 架构 | **并行**双路径，两个评分独立产生后**融合** |
| 抽样 | 结构化: 半随机分块抽样; 非结构化: 基于元数据的聚类抽样 |
| 后处理 | 角色判定 + 虚假数据过滤 + 单向函数标记 |
| 安全应用 | 基于分类结果修改云组件配置 |

**本方案对应内容:**

| 要素 | 本方案 | 是否相同 |
|------|--------|:---:|
| PII检测方法 | Tier 0 正则+字典规则引擎 (非真值表) | **不同** |
| NER方法 | DeBERTa-v3 encoder-only 判别模型 (非ML字符串分类器) | **不同** |
| 架构拓扑 | **串行** Tier 0→1→2→3 递进管道 | **不同** |
| 融合策略 | 无融合——Tier之间是串行传递，上游结果作为下游特征 | **不同** |
| LLM | Mistral-7B作为簇级分类和仲裁 (专利无LLM) | **不同** |
| 安全应用 | 无云组件修改功能 | **不同** |

**核心差异：并行融合 vs 串行递进。** 这是架构层面的根本不同。专利的权利要求围绕"两个路径产生两个独立评分 → 融合判断"展开。本方案是"规则提取特征 → 聚类压缩 → 多层分类 → 质量门"的串行管道。法律上，串行管道不落入并行融合权利要求的字面范围。

**结论: 不侵权。** 架构拓扑（串行 vs 并行融合）是本质差异。本方案使用的DeBERTa、BGE-M3、Mistral-7B等模型均未被专利权利要求覆盖。Tier 0的规则引擎与专利的真值表在技术实现上也不同（特征提取 vs 评分输出）。

### 专利五: US 12,316,686 B1 — 安全策略应用 (已授权)

**权利要求的核心要素:**
- 从多个不同网络安全平台接收多种格式的策略
- 使用生成式AI (LLM/SLM) 将策略归一化为统一表示
- 从归一化策略生成适用于其他平台格式的策略

**本方案对应内容:** 无。本方案不涉及策略管理、策略归一化、跨平台策略转换。

**结论: 不侵权。**

### 专利申请六~八: 聚类分类家族 — 最关键的风险分析

**状态: 已公开但尚未授权** (US 2024/0362301 A1, US 2025/0068701 A1, WO 2024/224367 A1)

由于尚未授权，最终权利要求范围未知。以下分析基于当前公开的说明书和审查历史（prosecution history）的合理推断。

#### 权利要求的核心要素（基于公开说明书）

| 步骤 | 专利描述的技术要素 |
|------|-------------------|
| **步骤1: 元数据替换** | 数值→替换为包含原值的预定义范围（如1223→"1024-2048"）；文本模式（标识符、日期、IP、UUID、十六进制）→替换为标准占位符；文本参数通过分隔符拆分为子串 |
| **步骤2: 聚类形成** | 基于**相同的替换后元数据**形成初始聚类；随机序列识别和替换；低于阈值的聚类进行文本参数合并 |
| **步骤3: 真实聚类判定** | 从聚类中抽样→分类→统计检验分类一致性→判定"真实聚类"或"虚假聚类" |
| **步骤4: 虚假聚类处理** | 基于元数据和分类差异拆分为子聚类→迭代重新处理 |
| **步骤5: 角色发现** | 从列名中提取重复术语→分析字段内容验证→创建新角色分类 |

#### 本方案 vs 专利申请 — 要素逐一对比

**步骤1: 聚类方法**

| | 专利申请 | 本方案 | 是否相同 |
|---|---|---|---|
| 核心机制 | **元数据值替换**（值→范围/占位符） | **结构特征提取+哈希**（提取文件类型、大小、页数、路径深度等特征→哈希） | **不同** |
| 数值处理 | 替换为包含原值的范围 | 作为结构特征之一（如文件大小分位数），不替换原值 | **不同** |
| 文本处理 | 模式识别→替换为占位符 | 不操作文本内容 | **不同** |
| 聚类依据 | 替换后的元数据是否**相同** | 结构特征哈希是否**相同** | 概念相似但机制不同 |

**关键区别：** 专利的方法是**修改数据本身**（替换值→然后基于相同替换值聚类）。本方案是**提取特征**（提取高层结构特征→哈希→分桶），从不修改原始数据内容。这是两种本质上不同的技术路径。

**步骤2: 嵌入和语义分析**

| | 专利申请 | 本方案 |
|---|---|---|
| 第二阶段聚类 | 无语义嵌入阶段 | Stage B: BGE-M3嵌入 + FAISS语义精炼 |
| 技术手段 | 纯元数据驱动 | 神经网络嵌入驱动 |

本方案有整个语义精炼阶段（嵌入→聚类），这是专利申请完全没有的技术维度。换言之，本方案在专利申请的基础上做了不同方向的技术延伸。

**步骤3-4: 真实/虚假聚类判定**

| | 专利申请 | 本方案 |
|---|---|---|
| 验证方法 | 抽样+分类+统计检验 | Tier 2 Step 5: 抽样+LLM验证 |
| 不一致处理 | 拆分子聚类+迭代重处理 | 标记"需拆分重聚" |
| 传播条件 | 统计检验通过→传播 | 不一致率<15%→传播 |

这是概念层面上最接近的部分——两者都使用"抽样+验证+传播"策略。但这是数据库和ML领域的通用工程实践（属于"cluster-then-classify"范式的常规步骤），不太可能被某一家独占。具体实现方式也不同：专利用统计检验，本方案用LLM交叉验证+阈值。

**步骤5: 角色发现**

| | 专利申请 | 本方案 |
|---|---|---|
| 方法 | 从列名提取重复术语 | 离群缓冲区→周期性重聚类→LLM描述新类型 |
| 适用性 | 仅结构化数据（有列名） | 结构化和非结构化数据 |

**结论: 即使该申请最终被授权，本方案的字面侵权风险较低。** 核心差异在于：
1. 本方案不使用元数据值替换——这是专利申请步骤1的核心机制
2. 本方案使用神经网络嵌入做语义聚类——这是专利申请未涉及的技术维度
3. 聚类+抽样+传播是通用工程范式，且本方案的具体实现方式不同

**需关注的潜在风险:** 如果USPTO授予该申请一个**宽泛的功能性权利要求**（如"A method comprising: clustering data objects based on their metadata characteristics; sampling from clusters; classifying samples; propagating classification to cluster members"），风险会上升。但基于该申请的说明书内容（极其具体地描述了元数据替换机制），审查员通常会要求权利要求限缩到说明书支持的具体范围。此外，已有公开的现有技术（如Google的MinHash聚类、Amazon Macie的分类方法）可能限制该申请的权利要求广度。

## 四、降低风险的现有技术防御

以下公开的现有技术和通用实践，如果Cyera试图主张宽泛的权利要求，可作为无效抗辩的依据：

1. **Google MinHash + LSH 聚类** (2014+): 基于特征哈希的文档聚类，早于Cyera的专利申请（2023年4月）
2. **Amazon Macie**: 使用ML + 规则进行数据分类，2017年发布
3. **Microsoft Presidio**: 开源PII检测框架，2018年发布，包含正则+ML+上下文分析
4. **"Cluster-then-Classify"范式**: 学术文献中广泛讨论，可追溯至2000年代初期
5. **Scikit-learn `FeatureHasher`**: 特征哈希的通用实现，早于Cyera专利
6. **FAISS (Facebook AI Similarity Search)**: 2017年发布，嵌入聚类的基础工具
7. **BERT/DeBERTa NER**: 基于encoder-only Transformer的NER，2020年后的学术主流

## 五、结论

**总体评估: 侵权风险较低。**

本设计方案与Cyera专利存在以下**根本性技术差异**，使其在字面侵权分析中不太可能落入任何已授权专利的权利要求范围：

1. **串行管道 vs 并行融合** — 与US12299167的核心架构差异
2. **结构特征哈希 vs 元数据值替换** — 与聚类分类专利申请的核心机制差异
3. **神经网络嵌入聚类 vs 纯元数据驱动聚类** — 增加了专利申请未涉及的技术维度
4. **不涉及数据发现和策略管理** — 与5项专利/申请完全无交集

**需要持续关注的:**

- US 2024/0362301 A1 的最终授权范围（预计2026-2027年）
- 该申请进入的国家阶段（中国CN、欧盟EP、日本JP等）的审查结果
- Cyera是否提交了新的延续申请（Continuation）或分案申请（Divisional），试图扩展权利要求范围以覆盖更宽泛的聚类分类方法

**建议的工程防御措施:**

1. 保留本方案各阶段的技术设计文档和决策记录，证明独立开发
2. 如需商业化，在正式发布前委托专利律师做正式的 **freedom-to-operate (FTO)** 分析
3. 关注US 2024/0362301 A1在USPTO的审查进展（可通过Public PAIR查询）
4. 避免在文档中将本方案的任何组件描述为"元数据替换"、"真值表"等Cyera专利术语
