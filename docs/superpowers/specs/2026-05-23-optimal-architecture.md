# 生产级文档分类架构设计

> 定位: 企业安全产品 (DSPM/DLP) 的文档智能分类引擎
> 架构核心: 多引擎并行融合 (借鉴 Cyera), 无单点故障
> 设计原则: 带着知识上线, 不确定时保守, 可观测可审计, 渐进退化

---

## 零、设计原则

| # | 原则 | 含义 |
|:---:|------|------|
| P1 | **带着知识上线** | 出厂内置规则和模型, 首日即可分类 |
| P2 | **不确定时保守** | 低置信度 → manual_review, 不臆断 |
| P3 | **可观测可审计** | 每个决策可追溯到每个引擎的输出 |
| P4 | **稳定优于智能** | 分类行为不自动漂移, 变更需证据 |
| P5 | **渐进退化** | 任一引擎故障不导致系统不可用 |

---

## 一、硬性要求

| # | 要求 | 量化标准 |
|:---:|------|---------|
| R1 | **多场景稳定** | 混合格式、同构文本, 两场景 Macro F1 差值 <5% |
| R2 | **准确率** | 正常运行时, 端到端 Macro F1 > 90% |
| R3 | **首次部署可用** | 出厂预置知识覆盖常见企业数据类型; 首篇文档即可分类 |
| R4 | **成熟期效率** | LLM 调用率 < 20%; 加权平均延迟 < 300ms |
| R5 | **不依赖数据假设** | 任一引擎失效时, 融合自动重分配权重; 不因数据特征突变而崩溃 |
| R6 | **异常检测与告警** | 分类质量下降 >5% 时告警; 低置信度 → manual_review; 全决策可审计 |
| R7 | **可证伪** | 每个模块有量化的失败条件; 每个条件可独立测试 |

### R2 的准确定义

正常运行时指所有引擎可用。任一引擎不可用时, 系统进入降级模式, 标记 degraded=true, R2 不适用降级模式。

### R3 的准确定义

"覆盖"指系统出厂自带的引擎（正则规则、预置模板）能对常见类型文档给出有效分类信号。不要求首日达到 90% F1（那是 R2 正常运行的指标），但要求首日可用——即至少有部分文档不需要 LLM 就能分类。

## 二、架构核心: 多引擎并行融合

### 2.1 为什么不用串行门控

```
串行门控 (上一版):
  信号评分 → composite≥0.85 → 直达 ┐
                                 ├→ 单阈值决定是否调LLM
              composite<0.85 → LLM ┘
  
  致命缺陷:
    阈值来自哪里? 预设 0.85 没有数据支撑
    直达采纳的文档对错无人知晓 → 错误永久化
    LLM 是不可替代的最终兜底 → LLM 挂了系统报废

并行融合 (本版):
  6 个引擎同时运行 → 各自输出 (标签, 置信度)
  → 融合 → 最终标签 + 综合置信度
  
  优势:
    没有"门控阈值"这个单点
    每个引擎的准确率独立可测
    任一引擎故障 → 其余引擎自动接管
    LLM 是 6 个引擎之一, 不是唯一兜底
```

### 2.2 六个引擎

```
┌──────────────────────────────────────────────────────────────────────┐
│ 引擎           │ 类型         │ 输出            │ 延迟   │ 依赖       │
├────────────────┼──────────────┼─────────────────┼────────┼────────────┤
│ E1: 正则规则   │ 确定性       │ 文档类型+置信度 │ <1ms   │ PII规则库  │
│ E2: 模板Hash  │ 确定性       │ 文档类型+置信度 │ <1ms   │ PII检测    │
│ E3: ML分类器  │ 统计         │ 文档类型+置信度 │ ~2ms   │ 训练数据   │
│ E4: 语义kNN   │ 统计         │ 文档类型+置信度 │ ~2ms   │ BGE-M3嵌入 │
│ E5: 结构签名  │ 确定性       │ 文档类型+置信度 │ <1ms   │ 结构特征   │
│ E6: LLM       │ 生成式       │ 文档类型+置信度 │ ~1.4s  │ Ollama服务 │
└──────────────────────────────────────────────────────────────────────┘

引擎独立性 (解决上一版的信号耦合问题):

  正则规则引擎: 仅依赖规则库, 不依赖PII检测结果
    (规则同时匹配文档文本和PII模式, 自我包含)
  
  模板Hash引擎: 依赖PII检测 → 实体替换 → Hash匹配
    如果PII检测故障 → 此引擎无输出 (不是错输出)
  
  ML分类器: 依赖SetFit模型文件, 独立于其他引擎
    未训练时 → 无输出
  
  语义kNN: 依赖BGE-M3嵌入 + 类型库质心
    BGE-M3故障 → 无输出
  
  结构签名: 依赖文件元数据提取 (pymupdf, python-docx等)
    元数据提取故障 → 无输出
  
  LLM: 仅依赖Ollama服务
    Ollama故障 → 无输出

关键: 所有引擎的依赖是独立的。
      不是"三个信号共用一个PII检测"。
      PII检测故障 → 只有 E2 受影响, E1/E3/E4/E5/E6 不受影响。
```

### 2.3 融合机制

```
融合策略: 加权投票 (与 Cyera 专利的"真值表融合"本质不同)

每个引擎对每个文档输出:
  (label, confidence, engine_id)

融合步骤:
  1. 按 label 分组, 将相同 label 的置信度加权求和
     score(label) = Σ (engine_weight × confidence × engine_available)
  
  2. 选择 score 最高的 label 作为最终标签
  
  3. 综合置信度 = max_score / Σ(所有engine权重)
     (归一化到 [0,1], 反映引擎间的一致性程度)

  4. 综合置信度 < 0.4 → 标记 manual_review
  
  5. 所有引擎均无输出时 → label="unclassified", manual_review=true
     (极端边缘情况: 所有引擎同时故障或无规则/模板/模型/质心/LLM可用)

引擎权重 (预置, 基于每个引擎的独立验证准确率):
  ┌────────────────────────────────────────────────────┐
  │ 引擎     │ 权重 │ 依据                             │
  ├──────────┼──────┼──────────────────────────────────┤
  │ E1 正则  │ 1.0  │ 确定性规则, 误报率低但覆盖面有限  │
  │ E2 模板  │ 1.0  │ 确定性Hash, 匹配即高置信          │
  │ E3 ML    │ 1.5  │ 统计模型, 覆盖面广, 置信度校准后  │
  │ E4 kNN   │ 1.0  │ 语义相似, 覆盖面最广              │
  │ E5 结构  │ 0.8  │ 仅区分文件格式, 不区分内容类型    │
  │ E6 LLM   │ 2.0  │ 最高准确率, 但延迟最高           │
  └──────────┴──────┴──────────────────────────────────┘

  权重设计逻辑:
    - LLM 权重最高 (2.0) → 当 LLM 有明确意见时, 优先采纳
    - ML (SetFit) 权重 1.5 → 蒸馏后的分类器, 覆盖面广
    - 确定性引擎 (E1/E2) 权重 1.0 → 精确但覆盖面有限
    - 结构签名权重 0.8 → 区分力弱, 辅助信号
    - 所有引擎可用时, LLM 主导决策 → 准确率接近 LLM 水平
    - LLM不可用时, 其余引擎融合 → 准确率下降但系统可用

置信度定义:
  每个引擎的置信度不是 softmax 概率, 而是该引擎对其输出的确信度:
  
  E1 正则: 置信度 = 匹配的规则条数 / 该类型期望规则条数 (截断到 1.0)
  E2 模板: 置信度 = 1.0 (精确Hash匹配) 或 0.5 (部分匹配)
  E3 ML:   置信度 = SetFit predict_proba 的最大值
  E4 kNN:  置信度 = 最近质心的余弦相似度
  E5 结构: 置信度 = 1.0 (精确匹配) 或 0.0 (不匹配)
  E6 LLM:  置信度 = LLM 输出的 confidence 字段

  置信度都是引擎自己的内部指标, 未经跨引擎校准。
  融合时通过权重平衡各引擎的话语权。
```

### 2.4 LLM 调用优化 (解决 R4)

```
LLM (E6) 不是默认激活的。激活策略:

  1. 首先运行 E1-E5 (总延迟 <5ms)
  2. 融合 E1-E5 的结果 → preliminary_label + preliminary_confidence
  3. 如果 preliminary_confidence ≥ 0.85:
     → 不调 LLM, 直接输出 (method="fusion_fast")
  4. 如果 preliminary_confidence < 0.85:
     → 调 LLM, 6引擎完整融合 (method="fusion_full")

  正常情况下, 预期:
    ~50% 文档 → preliminary_confidence ≥ 0.85 → 不调LLM (E1-E5 已达成共识)
    ~30% 文档 → preliminary_confidence < 0.85 → 调LLM
    ~20% 文档 → E1-E5 无引擎输出 → 直接调LLM

  
  为什么这不是"串行门控"?

  串行门控: 信号评分 → 阈值 → 直达或LLM (错了就错了)
             LLM 是兜底, 没有LLM就崩溃

  本方案:   E1-E5 先投票 → 如果已有共识 → 采纳
            → 如果没有共识 → 加入 LLM 再投票
            LLM 是额外一票, 不是兜底
            LLM 不可用时 → 就按 E1-E5 的初步融合结果输出
            (标记 degraded=true, 综合置信度可能较低)
```

## 三、各引擎详细设计

### 3.1 E1: 正则规则引擎

```
预置规则库 (出厂自带):
  50+ 文档类型推断规则, 每条规则:
    - 正则模式 (匹配文档文本中的关键词/模式)
    - 关联的 PII 类型 (可选, 增强置信度)
    - 目标文档类型

示例规则:
  Rule "HR_PAYROLL":
    pattern: (?i)(w-2|w2|payroll|salary|compensation|pay\s*stub)
    associated_pii: [SSN, EMPLOYER_ID]
    → label: "HR & Payroll", base_confidence: 0.7
  
  Rule "MEDICAL_RECORD":
    pattern: (?i)(diagnosis|patient|prescribed|medical\s*record|HIPAA)
    associated_pii: [MEDICAL_RECORD, NPI, DATE_OF_BIRTH]
    → label: "Medical Record", base_confidence: 0.7

  Rule "FINANCIAL_REPORT":
    pattern: (?i)(quarterly\s*report|annual\s*report|revenue|net\s*income|balance\s*sheet)
    associated_pii: [MONEY, CREDIT_CARD]
    → label: "Financial Report", base_confidence: 0.7

置信度计算:
  基础分 = base_confidence (规则命中)
  PII加成 = min(匹配的associated_pii类型数 / 期望类型数, 1.0) × 0.3
  最终置信度 = min(基础分 + PII加成, 1.0)

覆盖面: 50+ 规则覆盖 约 40-60% 常见企业文档类型。
        覆盖面窄于其他引擎, 但误报率最低。
```

### 3.2 E2: 模板Hash引擎

```
直接使用 Cyera 的元数据替换方案。

流程:
  1. Tier 0 PII检测 → 实体列表 → 类型占位符替换
     "SSN: 123-45-6789, Salary: $95,000" → "SSN: [SSN], Salary: [MONEY]"
  2. SHA256(替换后文本) → 模板Hash
  3. 与预置模板库和已知类型库中的模板Hash精确匹配
  4. 匹配到 → 输出对应文档类型, 置信度=1.0

预置模板库 (出厂):
  基于常见企业文档样本预计算。每个模板:
    template_hash: SHA256(PII替换后的样本文本)
    label: 文档类型
    sample_count: 用于生成此模板的样本文档数

PII稀疏处理:
  PII实体 < 3 → 替换后文本≈原文
  → 模板Hash匹配率低
  → 此引擎自然输出少 → 其他引擎接管
  → 不特殊处理, 让融合自然解决

覆盖面: 取决于客户数据中PII丰富的文档比例。
        HR/财务/医疗文档 → 高覆盖率
        通用文档 → 低覆盖率
```

### 3.3 E3: ML分类器 (SetFit)

```
训练触发: 任一类型 sample_count ≥ 50

推理:
  SetFit模型.predict(text) → label
  SetFit模型.predict_proba(text) → confidence

未训练时: 此引擎无输出。
训练后: 覆盖面广 (~80%+), 延迟 ~2ms。

与 V2.2 的差异:
  V2.2: SetFit 是分类加速器, 低置信回退LLM
  本方案: SetFit 是独立引擎, 和其他引擎平等投票
```

### 3.4 E4: 语义kNN

```
流程:
  1. BGE-M3嵌入文档 → embedding (1024维)
  2. 计算到所有已知类型质心的余弦距离
  3. 找最近质心 → 对应类型标签
  4. 置信度 = 1 - distance (余弦距离)

激活条件: 类型库 ≥ 5 个类型
覆盖面: 最广, 但置信度可能较低 (质心质量依赖样本积累)

质心更新:
  每个新标注样本 → 增量更新对应类型的质心
  质心 = (old × count + new_embedding) / (count + 1)
```

### 3.5 E5: 结构签名

```
流程:
  1. 提取文档元数据 (文件类型/大小分位/页数/段落数等)
  2. SHA256 → 结构签名
  3. 匹配已知类型的结构签名库

覆盖面: 窄。仅区分文件格式, 不区分内容类型。
        此引擎权重最低 (0.8), 主要作用是:
        - 区分 PDF vs DOCX vs JSON (格式级别)
        - 辅助其他引擎 (结构签名一致 → 增强了其他引擎的可信度)

注意: 此引擎和 E2 的区别:
  E2: PII替换后的内容Hash → 区分内容模式
  E5: 文件元数据Hash → 区分格式
  两者互补, 不是重复。
```

### 3.6 E6: LLM

```
模型: Mistral-7B (Ollama, Q4)
激活: preliminary_confidence < 0.85 时 (见 §2.4)
延迟: ~1.4s/文档

LLM 提示 (保持与DataDNA一致的XML标签结构):
  <instruction>Classify the document into a document type.</instruction>
  <document>[PII替换后的文本, 前2000字符]</document>
  <output_schema>{"label": "...", "confidence": 0.0-1.0, 
   "is_new_type": false, "rationale": "..."}</output_schema>

LLM 不可用时的降级: 见 §6。
```

## 四、融合示例

```
文档: "Employee SSN: 123-45-6789, Name: John Smith, Salary: $95,000"

引擎输出:
  E1 正则:  ("HR & Payroll",  0.85)  ← 命中 HR_PAYROLL 规则 + SSN加成
  E2 模板:  ("HR Record",     1.00)  ← 模板Hash精确匹配
  E3 ML:    无输出                    ← SetFit未训练
  E4 kNN:   ("HR & Payroll",  0.72)  ← 最近质心是HR类型, 距离0.28
  E5 结构:  ("unknown",       0.00)  ← 无匹配的结构签名
  E6 LLM:   未调用                    ← E1-E5 融合分数已够

融合:
  score("HR & Payroll") = 1.0×0.85 + 1.0×0.72     = 1.57  (E1+E4)
  score("HR Record")    = 1.0×1.00                 = 1.00  (E2)
  score("unknown")      = 0.8×0.00                 = 0.00  (E5)
  
  → 最终标签: "HR & Payroll"
  → 综合置信度: 1.57 / (1.0+1.0+1.0) = 0.52
  
  综合置信度 0.52 ≥ 0.4 → 不标记 manual_review
  method: "fusion_fast" (LLM未调用)
```

```
文档: "The weather is nice today and the sun is shining brightly."

引擎输出:
  E1 正则: 无输出      ← 无规则命中
  E2 模板: 无输出      ← PII=0, 无模板匹配
  E3 ML:   无输出      ← 未训练
  E4 kNN:  ("General", 0.35) ← 最近质心距离远
  E5 结构: 无输出
  → preliminary_confidence < 0.85 → 调LLM

  E6 LLM:  ("General / Non-Business", 0.60)
  
融合 (6引擎):
  score("General / Non-Business") = 2.0×0.60 = 1.20
  score("General")                = 1.0×0.35 = 0.35
  
  → 最终标签: "General / Non-Business"
  → 综合置信度: 1.20 / (1.0+2.0) = 0.40
  
  综合置信度 0.40 ≥ 0.4 → 刚好不标记 manual_review
  LLM 置信度 0.60 < 0.7 → 考虑标记 manual_review
  
  实际上 0.40 边界值 → 标记 manual_review (保守原则)
  method: "fusion_full"
```

## 五、类型库生命周期

```
类型来源:
  1. 出厂预置 (builtin): 50+ 类型, 带规则/模板/关键词
  2. LLM 新类型: LLM is_new_type=true → 注册
  3. 类型发现: 聚类检测到的新模式 → 注册

类型结构:
  {
    type_id: "hr_payroll",
    type_name: "HR & Payroll",
    source: "builtin",
    rules: [E1规则列表],
    template_hashes: [E2模板Hash列表],
    pii_distribution: {"SSN": 0.8, "EMAIL": 0.3, ...},
    semantic_centroid: np.array(1024),
    keywords: ["payroll", "salary", "W2", ...],
    structural_signatures: [hash1, hash2, ...],
    sample_count: 150,
    status: "active"
  }

类型去重:
  新类型注册时与已有类型比较:
    名称相似度 > 0.8 (编辑距离) + 语义距离 < 0.15 → 合并
    合并需运维确认 (安全产品不自动合并类型)

类型过期:
  builtin: 永不过期
  动态类型: 180天无新样本 → deprecated (仍可用)
           365天无新样本 → 运维确认后删除
```

## 六、降级路径

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 故障引擎 │ 降级行为                │ 影响                          │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E1 正则  │ 此引擎无输出            │ 微小。其他引擎覆盖类似类型     │
│          │ 融合忽略此引擎          │                               │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E2 模板  │ 此引擎无输出            │ 小。E1/E3/E4/E6 可接管        │
│          │                         │ PII丰富文档失去一个强信号     │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E3 ML    │ 此引擎无输出            │ 小-中。效率下降(无SetFit加速) │
│          │                         │ 但不影响准确率                │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E4 kNN   │ 此引擎无输出            │ 中。覆盖面最广的引擎失效      │
│ BGE-M3   │ 类型发现暂停            │ 新类型无法发现(手动注册仍可)  │
│          │ 质心更新暂停            │ 已有质心仍可用于匹配          │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E5 结构  │ 此引擎无输出            │ 微小。此引擎权重最低          │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ E6 LLM   │ 此引擎无输出            │ 中-大。准确率可能下降          │
│          │ 融合仅用E1-E5           │ 但系统仍可用                  │
│          │ 综合置信度可能偏低      │ manual_review可能增多         │
│          │ 所有文档标记degraded    │ 边缘文档可能分类不准确        │
├──────────┼─────────────────────────┼───────────────────────────────┤
│ 所有引擎 │ E1+E5 确定性引擎仍可用  │ 确定性引擎覆盖面窄但准确      │
│ 同时故障 │ 其余文档标记unclassified│ 系统基本可用, 但分类率低      │
│ (极端)   │ (不是系统崩溃)          │                               │
└──────────┴─────────────────────────┴───────────────────────────────┘

关键原则:
  E6 (LLM) 不可用时, 系统不崩溃。E1-E5 继续融合。
  这是并行架构相比串行LLM兜底的根本优势。
```

## 七、质量监控

```
每个文档处理后, 记录:
  - 哪些引擎有输出
  - 各引擎的 (label, confidence)
  - 融合后的 final_label + composite_confidence
  - method: fusion_fast | fusion_full
  - degraded: true/false
  - manual_review: true/false

质量校验:
  fusion_fast 文档抽样 2% 送 LLM 做一致性校验
  (fusion_fast 跳过了 LLM, 需要验证 E1-E5 共识是否可靠)
  校验不一致率 > 5% → 告警 "E1-E5共识质量下降"

聚合监控 (每小时):
  1. 融合置信度分布 → 告警: P50 < 0.3
  2. 各引擎输出率 → 告警: 任一引擎输出率骤降 >30%
  3. LLM调用率 → 告警: >50% (效率异常) 或 =0% (可能LLM挂了)
  4. manual_review 积压 → 告警: >10%
  5. 新类型注册速率 → 告警: >10/小时 (可能数据分布突变)
  6. 标签分布 KL散度 vs 昨日 → 告警: >0.3 (可能数据漂移)
  7. fusion_fast校验不一致率 → 告警: >5%

不自动修复。告警 → 运维决策。
```
