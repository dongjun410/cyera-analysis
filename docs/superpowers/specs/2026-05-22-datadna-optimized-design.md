# DataDNA 分类引擎 — 工程最优实现方案

> 不追求复刻Cyera架构，基于工程最佳实践独立设计
> 设计日期：2026-05-22
> 修订日期：2026-05-22 (v1.2 — FP16→INT8，适配12GB消费级显卡)

---

## 一、设计原则

1. **模型选型匹配任务性质，不匹配厂商选型。** NER是判别任务，用判别模型。语义理解是生成/推理任务，用生成模型。
2. **聚类是架构核心，不是预处理。** 对簇分类而非对文档分类，是吞吐量的根本来源。
3. **快路径处理大多数，慢路径处理少数。** LLM是稀缺资源，仅用于无法被规则和轻量模型解决的边界问题。
4. **类型发现是聚类+离群检测的自然产物，不需要独立架构。**
5. **简单优于复杂。** 两条路径够用就不设三条，一个LLM够用就不部署两个。

**数据流澄清：** Tier 0 不产生最终分类决策——它产生的是 PII 特征向量（检测到的实体类型、位置、置信度、上下文标记），注入后续所有 Tier。全部文档（100%）流经 Tier 0 获取 PII 特征，然后全部文档进入 Tier 1 做嵌入和聚类。Tier 0 的"匹配已知 PII 模式"用于：(a) 作为 Tier 2 已知类型匹配的输入特征之一，(b) 高置信度 + 无上下文惩罚词的匹配结果直接标记为确定性 PII 实体，但文档本身的类别归属仍由 Tier 2 决定。

## 三、模型选型及理由

### 3.1 嵌入模型: BGE-M3

| 属性 | 值 | 理由 |
|------|-----|------|
| 维度 | 1024 | 质量与存储的平衡点 |
| 上下文 | 8192 token | 覆盖绝大多数企业文档 |
| 语言 | 100+ | 多语言企业环境必需 |
| 推理 | ~2ms/文档(GPU) | 10万文档嵌入 ~3分钟 |

**为什么不是FLAN-T5 Encoder:** T5系列encoder的隐藏状态针对解码器交叉注意力优化，训练目标是next-token prediction而非语义相似度。BGE-M3使用对比学习训练，天然产出适合余弦相似度比较的嵌入空间。这是训练目标的根本差异——不是"FLAN-T5 encoder不好"，而是"它被训练来解决不同的问题"。

**为什么不是e5-mistral:** e5-mistral在英文上表现优异，但多语言覆盖不如BGE-M3。企业文档的多语言需求决定了BGE-M3是更安全的选择。

### 3.2 NER/PII检测: DeBERTa-v3-base (微调)

| 属性 | 值 | 理由 |
|------|-----|------|
| 参数量 | 184M | 轻量，CPU可推理 |
| 推理速度 | ~5ms/文档(CPU), ~1ms(GPU) | 比生成式NER快50-100倍 |
| 架构 | Encoder-only + token分类头 | NER是判别任务，判别模型是最优解 |

**为什么不是FLAN-T5做NER:** 这是一个延迟与精度的权衡，不是精度的绝对优势。

- FLAN-T5微调在CoNLL-03上可达 **96.14% Macro F1**（`pepegiallo/flan-t5-base_ner`），这个数字高于DeBERTa-v3在同等条件下的 ~93% F1。如果精度是唯一指标，FLAN-T5胜出。
- 但FLAN-T5做NER需要decoder逐token生成标签序列——encoder一次前传 + decoder N次前传（N = 序列长度）。DeBERTa单次前传完成全部token的分类。在批处理场景中，DeBERTa的延迟是FLAN-T5生成式NER的 **1/50 ~ 1/100**。
- **决策逻辑：** 在分类管道中，NER作为中间特征提取步骤，延迟比精度关键——NER结果会被Tier 2的已知类型匹配和LLM进一步精炼，NER阶段3个点的F1差距可以在下游被补偿。如果NER是最终输出（如直接面向用户的PII检测工具），FLAN-T5更合适。

### 3.3 LLM: Mistral-7B-Instruct (INT8用于质量门, 4-bit用于批量分类)

| 属性 | 值 | 理由 |
|------|-----|------|
| 参数量 | 7B | 本地可部署的最小实用LLM |
| 调用场景 | 簇命名 + 低置信度仲裁 + 新型号描述 | 仅处理约5-10%的数据 |
| Tier 2 推理 | 4-bit QLoRA (batch推理, 吞吐优先) | ~5GB VRAM |
| Tier 3 推理 | INT8 (质量门, 精度优先) | ~8-9GB VRAM, 消费级12GB显卡可承载 |

**量化策略分层的理由:** 4-bit量化在分类准确率上相比INT8下降1-3个百分点。Tier 2的LLM调用量大但容忍一定的精度损失（结果会被Tier 3验证），用4-bit。Tier 3是最终仲裁者，需要更高精度，用INT8。INT8相比FP16精度损失通常在1个百分点以内，但显存需求从~14GB降至~8-9GB，使12GB消费级显卡（RTX 5070/4070/3080等）也能承载。同一模型，不同量化配置，按场景选择。

**为什么不是FLAN-T5做语义:** FLAN-T5-large只有780M参数，语义理解能力远不如7B decoder-only模型。在需要深层推理的场景（判断文档的业务敏感性、区分客户vs员工数据、识别新型数据类型的语义），参数量的差距无法通过微调弥补。

**为什么不是更大模型:** 7B在INT8量化后可在消费级12GB显卡上推理。12B或更大的模型需要更多VRAM或更激进的量化，部署复杂度显著增加。

### 3.4 模型使用总览

| 模型 | 角色 | 处理比例 | 每次调用延迟 | 部署要求 |
|------|------|:---:|:---:|------|
| 正则+规则引擎 | PII特征提取 | 100% | <0.1ms | CPU |
| BGE-M3 | 文档嵌入(聚类用) | 100% | ~2ms | GPU推荐,CPU可用 |
| DeBERTa-v3-base | 上下文NER消歧 | ~15% | ~5ms | CPU/GPU |
| Mistral-7B (4-bit) | Tier 2 簇级批量分类 | ~5-10% | ~300ms | GPU (5GB) |
| Mistral-7B (INT8) | Tier 3 质量门仲裁 | ~1-2% | ~400ms | GPU (9GB) |

> 注：处理比例指"触发该模型处理的文档占比"，不是该模型处理了全部文档。BGE-M3嵌入全部文档是聚类的基础需求——没有嵌入无法计算语义距离和做Stage B精炼。

## 四、核心组件设计

### 4.1 Tier 0: 确定性规则引擎

**定位:** 快速PII特征提取层。注意：Tier 0产生的是**特征**，不是最终分类标签。即使检测到高置信度SSN模式，文档的最终类别（如"HR-含SSN的入职文件" vs "财务-含SSN的税务表"）仍由Tier 2决定。

```
规则结构:
  pattern: 正则模式
  entity_type: 实体类型
  validation: 校验函数 (Luhn算法/校验和/格式规范)
  context_boost_terms: [上下文增强词]  (如 "SSN:", "社会安全号")
  context_penalty_terms: [上下文惩罚词]  (如 "test", "sample", "example", "TODO")
  context_window: 前后N字符
  min_confidence: 最低置信度阈值
  output: PII实体列表 + 每个实体的(类型, 位置, 置信度, 上下文标记)

规则来源:
  - 内置: 50+ 常见PII模式 (基于Presidio + 自定义扩展)
  - 行业扩展: 按行业加载的额外规则包
  - 客户自定义: 通过配置文件或UI添加
```

**关键设计决策:** 当规则引擎检测到PII模式但上下文中存在penalty词时，不直接判定为误报，而是标记为"低置信度"并在PII特征向量中保留上下文标记。Tier 2的DeBERTa会消费这些特征做深度判别。Tier 0不直接丢弃任何信息——它只做特征化。

**输出数据结构:**
```
{
  "doc_id": "...",
  "pii_features": [
    {"type": "SSN", "span": [120, 131], "confidence": 0.95, "context_flag": "clean"},
    {"type": "CREDIT_CARD", "span": [450, 466], "confidence": 0.45, "context_flag": "penalty_term_present"}
  ],
  "pii_type_distribution": {"SSN": 1, "CREDIT_CARD": 1},
  "has_high_conf_pii": true,
  "has_penalty_terms": true
}
```

### 4.2 Tier 1: 双阶段聚类

**定位:** 架构的核心。将大量文档压缩为少量簇代表，后续所有分类操作针对簇而非文档。

#### Stage A: 结构聚类

```
目标: 按文档结构特征快速分桶

结构特征向量 (文档):
  - 文件类型 (.pdf, .docx, .xlsx, .json, etc.)
  - 文件大小分位数 (对数分桶)
  - 页数/段落数/表格数
  - 是否含图片
  - 页眉/页脚模式 (regex匹配)
  - JSON/XML schema签名 (如有)
  - 路径深度/父目录名

结构特征向量 (数据库列):
  - 数据类型 (string/numeric/date/bool)
  - 值长度分布 (min/max/mean/std)
  - Null比例
  - 唯一值比例
  - 字符集检测 (纯数字/字母/混合/含特殊字符)
  - 常见值模式聚类

聚类方法: 确定性哈希分桶
  - 对结构特征做确定性哈希
  - 相同哈希 → 同一结构簇
  - 无需K值，无需迭代，O(N)时间
  - 自动发现结构类型 (如"3列表格PDF"、"JSON API响应"等)

产出: ~100-1000个结构簇，代替原始N个文档
```

#### Stage B: 语义精炼

```
目标: 在每个结构簇内部，按语义相似度进一步分组

仅在以下条件触发:
  - 结构簇大小 > sem_split_threshold (建议50)

方法:
  1. 对簇内所有文档做BGE-M3嵌入
  2. 计算簇内嵌入的方差——如果簇内成对余弦相似度均值 > 0.85:
     → 高度同质，不需要拆分
  3. 如果簇内相似度标准差 > 0.25 (存在语义分层):
     a. 使用FAISS IVF索引对簇内嵌入建索引
     b. 使用基于密度的子簇发现 (HDBSCAN 或 FAISS KMeans + 轮廓系数自动选K)
     c. 拆分为语义子簇
  4. 对于超大簇 (>10,000文档):
     先随机采样10,000个文档做聚类结构探索
     → 确定子簇数量和中心
     → 将其余文档通过FAISS最近邻分配到最近的子簇中心
     → 时间复杂度: O(N log N) 而非 O(N²)

产出: 语义精炼后的最终文档簇
```

**可扩展性设计:** Stage B默认使用FAISS IVF索引 + HDBSCAN替代AgglomerativeClustering。FAISS IVF将搜索复杂度从O(N²)降到O(N log N)。对于10万文档的簇，传统成对相似度矩阵需要~40GB内存，FAISS IVF索引仅需~400MB。

#### 增量文档分配

```
新文档到达时的簇分配策略:

1. 提取新文档的结构特征 → 确定性哈希 → 定位到Stage A结构桶
2. 如果结构桶存在:
   a. BGE-M3嵌入新文档
   b. 计算新文档到桶内各语义子簇中心的余弦距离
   c. 如果最近距离 < 桶内簇半径的1.5倍 → 分配到该子簇
   d. 如果最近距离 > 1.5倍半径 → 标记为"桶内离群"
3. 如果结构桶不存在 (全新结构类型):
   → 创建新结构桶
   → 文档直接标记为"新型结构候选"
4. 触发条件 (满足任一):
   - 桶内离群文档累计 > 该簇大小的20%
   - 新结构桶内文档 > sem_split_threshold
   → 对该桶触发Stage B语义精炼重跑

时间复杂度: O(1) 结构哈希 + O(log K) FAISS检索 = 每文档 <5ms
```

**与V2.2 KMeans聚类和Cyera专利的区别:**

| | 本方案 | V2.2 | Cyera专利 |
|---|---|---|---|
| 第一阶段 | 结构哈希（确定性） | 无 | 元数据替换 |
| 第二阶段 | FAISS语义精炼（簇内） | KMeans（全局） | 无 |
| K值 | 自动（结构种类数×语义分层数） | 需搜索 | 自动（模式种类数） |
| 可解释性 | 高（可追踪到结构特征） | 低（嵌入是黑箱） | 中 |
| 增量友好 | 哈希定位 + FAISS检索 O(log N) | 需kNN或全量重聚类 | 模式匹配 O(1) |

### 4.3 Tier 2: 簇级分类

**定位:** 对簇的代表样本进行分类，结果传播到全簇。

```
输入: 聚类结果 (簇列表 + 每个簇的代表样本 + 簇特征 + Tier 0的PII特征向量)

对每个簇执行:

Step 1: 簇特征提取
  - 结构特征摘要 (来自Stage A)
  - TF-IDF关键词 Top-15
  - PII类型分布计数 (来自Tier 0的特征向量)
  - 语言分布

Step 2: 已知类型快速匹配
  对每个簇特征与已知类型库做相似度匹配:
    - 结构签名精确匹配 → 强信号 (权重0.5)
    - TF-IDF关键词重叠度 > 0.6 → 中等信号 (权重0.3)
    - PII类型分布余弦相似度 > 0.8 → 辅助信号 (权重0.2)
  综合评分 = 加权和
  综合评分 > 0.8 → 已知类型，无须LLM
  综合评分 [0.5, 0.8) → 进入Step 4 (LLM确认)
  综合评分 < 0.5 → 未知类型候选，进入Step 4

  已知类型库初始为空。冷启动时所有簇的综合评分为0，100%进入Step 4。
  随着系统运行，类型库逐步积累，Step 2匹配率从0%逐步上升到70-90%。

Step 3: DeBERTa NER扫描
  对簇代表样本执行PII检测:
    - 上下文识别 (区分真实PII vs 测试数据 vs 代码注释)
    - 边界精确定位
    - 输出: 实体类型 + 位置 + 置信度 + 上下文验证结果
    - 消费Tier 0的PII特征向量: 对Tier 0标记为"低置信度"的实体做深度判别

Step 4: LLM分类 (仅当Step 2未匹配, 或匹配评分在[0.5, 0.8))
  模型: Mistral-7B-Instruct (4-bit QLoRA, 吞吐优先)
  输入: 簇特征 + 代表样本文本 (前2000字符) + DeBERTa NER结果 + Tier 0 PII特征
  Prompt: 分类任务，给定已知类型列表 + "新类型"选项
  输出: {
    "label": "已知类型名 或 新类型候选名",
    "is_new_type": bool,
    "confidence": 0.0-1.0,
    "rationale": "一句话理由",
    "suggested_rules": "如果是新类型，建议的正则/结构特征"
  }

Step 5: 传播与验证
  - 簇标签 → 所有簇内文档继承
  - 高敏感簇 (含SSN/信用卡/医疗): 标记需要Tier 3二次确认
  - 簇内按比例抽样 (与簇大小成反比, 最少3篇) 做LLM验证
  - 验证结果与传播标签不一致率 > 15% → 整簇标记为"需拆分重聚"
```

**冷启动策略:**

```
Phase -1: 缓冲积累期

条件: 已知类型库为空 OR 文档总数 < min_clustering_threshold (建议50)

策略:
  1. 文档持续到达 → 写入文档缓冲区 (ES索引: cold_start_buffer)
  2. 对每个文档执行 Tier 0 (PII特征提取)
  3. 缓冲区内文档数 >= min_clustering_threshold:
     → 触发首次全量 Tier 1 聚类
     → 所有簇进入 Tier 2 Step 4 (LLM分类, 因为已知类型库为空)
     → LLM产出的类型标签注册为初始已知类型库
     → 进入正常运行模式
  4. 在缓冲积累期间到达的查询请求:
     → 对单个文档执行 Tier 0 + 零样本分类 (LLM直接判断, 不做聚类)
     → 标记为 "临时标签"
     → 系统进入正常运行模式后, 该文档被聚类吸收, 标签被簇标签覆盖

冷启动期间的延迟: 单文档LLM分类 ~500ms
正常运行后的延迟: 摊销到簇级, ~5ms/文档 (蒸馏后 ~2ms)
```

### 4.4 Tier 3: LLM质量门

**定位:** 最后的精度防线。仅处理少量高风险或低置信度的case。

```
模型: Mistral-7B-Instruct (INT8, 精度优先)

触发条件 (满足任一):
  1. 簇被分类为高敏感类型 (SSN/信用卡/医疗记录/金融凭证)
  2. 文档对簇中心的语义距离 > 2σ (离群文档 — 2σ约覆盖95%的正态分布文档)
  3. DeBERTa NER结果与Tier 0规则结果矛盾
  4. LLM分类置信度在 [0.5, 0.8) 区间
  5. 文档被Tier 2 Step 5标记为"需验证"

σ 阈值说明:
  - 2σ (Tier 3触发): 约5%的文档进入质量门审核。这些是需要LLM深度分析的"可疑"文档
  - 3σ (类型发现触发): 约0.3%的文档。这些是极大概率不属于任何已知类型的"完全离群"文档
  - 两个阈值的分工: 2σ触发"这属于哪个已知类型?"的审核; 3σ触发"这属于哪个新类型?"的探索

处理逻辑:
  对每个触发文档:
    ├── Mistral深度分析 (完整文档上下文 + 元数据)
    ├── 输出: 最终分类 + 置信度 + 推理链
    └── 如果置信度仍然低 → 标记为 "manual_review"

批量优化:
  - 同一簇的验证请求批量提交 (利用簇上下文增强)
  - 已验证文档作为簇内参考: "该簇中已验证的N篇中M篇为X类"
```

### 4.5 类型发现 (Learned Classification)

**定位:** 不是独立组件。是Tier 1离群缓冲区 + 周期性重聚类的自然产物。

```
流程:

1. 离群收集 (持续)
   来源:
     ├── Tier 2 Step 2: 无法匹配任何已知类型的簇 (综合评分 < 0.5)
     ├── Tier 2 Step 4: LLM标记为 is_new_type=true
     └── Tier 3: 文档语义距离 > 3σ 且不属于任何已知簇

   存储: 离群文档缓冲区 (ES索引: outlier_buffer)

2. 触发条件 (任一):
   ├── 离群缓冲区达到 N_min (建议100) 篇文档
   ├── 同一未知模式连续出现 M 次 (建议5次)
   └── 定时触发 (每24小时)

3. 新类型聚类
   对离群缓冲区执行Tier 1双阶段聚类
   → 产出候选新类型簇

4. 候选类型评估
   对每个候选簇:
     ├── 簇内一致性 > 阈值 (0.75)?
     ├── 与已知类型库的最近距离 > 阈值 (0.3)?
     ├── 簇大小 >= 最小新类型规模 (3)?
     └── LLM生成类型名称、描述、建议检测规则

5. 新类型注册
   通过评估 → 创建新类型记录:
     ├── type_id, type_name, type_description
     ├── 结构签名 (用于Tier 2 Step 2快速匹配)
     ├── 语义嵌入中心 (用于Tier 2 Step 2已知类型匹配)
     ├── 建议检测规则 (正则/结构特征/LLM prompt)
     └── 状态: "active" | "pending_review"

   新类型注册后:
     ├── 更新已知类型库 → Tier 2 Step 2匹配率提升
     ├── 触发蒸馏分类器重训练 (积累 >= 50 个新类型样本后)
     └── 新类型在蒸馏重训练前走Tier 2+3完整流程 (无加速)

6. 离群缓冲区清理
   已分类为新类型或已知类型的文档 → 移出缓冲区
   仍然无法分类 → 保留，等待更多样本
```

## 五、知识蒸馏

**定位:** 工程优化层。将Tier 2+3的LLM分类能力压缩到轻量分类器。

```
蒸馏时机:
  - 首次: 首次全量分类完成 + 已知类型库积累每类 >= 50 样本
  - 增量: 新分类样本累计 >= 500 或 发现新类型 (且该类型样本 >= 50)

教师: Tier 2+3产出的分类结果 (LLM + 规则 + NER的多源标签)
学生: SetFit (BGE-M3 base, ~100MB)

训练数据构建:
  来源: 所有最终分类结果 (包括LLM、规则匹配、质量门验证)
  筛选: 保留置信度 > 0.85 的样本
  人工审核: 每类随机抽取 10% (最少5个, 最多50个) 进行人工审核
    目的: 检测和修正管道系统性偏差, 防止蒸馏放大偏差
    如果某类人工审核一致率 < 80%:
      → 该类暂停蒸馏, 标记为 "review_required"
      → 提升该类样本进入Tier 3的比例
  采样: 每类型 50-200 样本 (分层, 少类多采, 多类少采)
  验证: 10% 留出

部署后调用链:
  新文档 → 蒸馏分类器 (~2ms, CPU)
    ├── 置信度 >= 0.85 → 采纳
    ├── 置信度 [0.6, 0.85) → kNN簇分配验证
    └── 置信度 < 0.6 → 回退到Tier 2+3完整流程

增量更新:
  触发: 新分类样本累计 >= 500 或 发现新类型
  策略: full retrain (数据量小, 训练快, 不需要增量微调)
  风险: full retrain可能导致旧类型分类性能退化 (catastrophic forgetting)
  缓解: retrain后必须在留出验证集上评估每个已知类型的F1, 
        任一类F1下降 > 3% → 回滚到上一版蒸馏模型
```

## 七、数据安全

### 7.1 LLM SaaS侧的敏感数据保护

当Mistral部署在SaaS侧时，Tier 2 Step 4和Tier 3需要将文档文本发送到外部。以下措施降低数据暴露风险：

1. **文档预处理 (在客户侧完成):** 在发送LLM之前，用Tier 0的PII检测结果对文本做脱敏替换。将检测到的PII实体替换为类型占位符 (`[SSN_1]`, `[CREDIT_CARD_1]`, `[EMAIL_1]`)。LLM在SaaS侧看到的是脱敏后的文本，但语义结构保留。
2. **截断控制:** 仅发送前2000字符给LLM，而非完整文档。对于长文档，用TF-IDF提取最关键的段落发送。
3. **传输加密:** TLS 1.3 + mTLS双向认证。
4. **LLM提供商选择:** 优选支持私有部署的提供商（Azure AI Foundry, AWS Bedrock），而非公共API（OpenAI public endpoint）。

### 7.2 Prompt注入防护

恶意文档可能包含prompt注入payload。防护措施：

1. **输入清洗:** 在文档文本拼入prompt前，检测并移除已知的注入模式（"ignore previous instructions", "system:", 等）
2. **结构化输出约束:** LLM输出由JSON schema约束（通过guidance库或outlines库），即使注入成功，输出格式也受限
3. **输入隔离:** 文档文本作为data block单独标记，与system prompt和instruction明确分隔：
   ```
   <system>You are a document classifier...</system>
   <instruction>Classify the following document:</instruction>
   <document>
   [客户侧脱敏后的文档文本]
   </document>
   <output_schema>{"label": "...", "confidence": 0.0-1.0, ...}</output_schema>
   ```
   文档内容被隔离在 `<document>` 标签内，LLM被训练为不将数据区内容解释为指令。

### 7.3 嵌入向量的隐私风险

BGE-M3嵌入向量可能通过成员推断攻击泄露原文信息。当嵌入存储在SaaS侧时：
- 嵌入向量**不离开客户侧**。Tier 1在客户侧完成嵌入和聚类，仅将簇元数据（簇ID、代表样本、PII特征）传递到下游。
- FAISS索引在客户侧构建和存储。

## 八、可观测性与运维

### 8.1 关键指标

| 指标 | 定义 | 告警阈值 |
|------|------|---------|
| **Tier 0 PII检测率** | 至少检测到1个PII实体的文档占比 | <10% 或 >90% (异常) |
| **Tier 1 聚类时间** | Stage A + Stage B 总耗时 | >10min (100K文档) |
| **Tier 1 簇数量** | 最终产出的簇总数 | <10 或 >5000 (异常) |
| **Tier 2 LLM触发率** | 进入Step 4的簇占比 | >30% (已知类型库可能失效) |
| **Tier 2 传播验证不一致率** | Step 5抽检不一致比例 | >15% (聚类质量下降) |
| **Tier 3 触发率** | 进入质量门的文档占比 | >10% (上游分类质量下降) |
| **蒸馏分类器准确率** | vs Tier 2+3完整管道的一致率 | <90% (蒸馏质量下降) |
| **端到端延迟 P95** | 文档从进入到产出的P95延迟 | >10s (实时场景) |
| **LLM调用成功率** | LLM请求成功/总请求 | <95% |
| **离群缓冲区增长率** | 每天新增离群文档数 | 突增>3倍 (数据分布变化) |

### 8.2 分类漂移检测

```
方法: 对每周新分类的文档, 计算以下分布差异:

1. 文档类型分布 (已知类型占比的变化)
   → KL散度 vs 上周分布 > 0.3 → 告警 "数据类型分布漂移"

2. 簇内平均语义距离趋势
   → 连续4周上升 → 告警 "文档内容漂移, 可能需要重聚类"

3. LLM触发率趋势
   → 连续2周上升超过10个百分点 → 告警 "已知类型库覆盖下降"

4. 蒸馏分类器回退率
   → 置信度 < 0.6 的文档占比 > 20% → 告警 "蒸馏模型退化, 需要重训练"
```

### 8.3 日志与追踪

每个文档的处理路径可追踪：`doc_id → Tier 0 耗时/PII数 → Tier 1 结构桶ID/子簇ID → Tier 2 匹配评分/方法(Step2/Step4)/置信度 → Tier 3 触发/未触发 → 最终标签`

日志以结构化JSON输出到ELK/Loki，支持按Tier、簇、类型、时间范围查询。

## 九、容错与降级

### 9.1 降级路径

| 故障场景 | 降级策略 | 影响 |
|---------|---------|------|
| BGE-M3嵌入服务故障 | 跳过Stage B语义精炼, 仅用Stage A结构聚类。簇变粗但管道不中断 | 分类粒度下降, 部分簇需Tier 3验证 |
| DeBERTa NER故障 | Tier 2 Step 3跳过, Step 4 LLM直接消费Tier 0 PII特征 | LLM调用量增加 ~10% |
| Mistral (4-bit) Tier 2故障 | Step 2匹配置信度降低阈值到0.5, 匹配到的直接采纳, 未匹配的标记"unclassified" | 部分文档无最终标签, 进入人工审核队列 |
| Mistral (INT8) Tier 3故障 | Tier 3触发条件1(高敏感簇) → 文档标记"pending_verification"暂不进ES索引; 其他触发 → 采纳Tier 2结果 | 高敏感分类结果暂不可用 |
| ES索引故障 | 分类结果写入本地SQLite降级存储, ES恢复后批量回放 | 查询功能暂时不可用, 分类功能正常 |
| 已知类型库损坏/丢失 | 所有簇回退到Tier 2 Step 4 (LLM分类), 从LLM输出重建类型库 | LLM调用量短期暴涨, ~1次/簇 |

### 9.2 重试与超时

- LLM调用: 最多重试2次, 指数退避 (1s, 2s), 超时30s
- BGE-M3嵌入: 批量重试, batch内单个失败不阻塞batch
- FAISS索引构建: 无重试 (确定性操作), 失败直接进入Stage B降级路径
- 聚类操作: 无重试 (确定性操作), 失败 → 所有文档标记"unclustered", 直接进入Tier 2逐文档分类 (降级模式)

## 十、端到端评估体系

### 10.1 评估数据集

```
构建方式:
  - 从客户文档中随机抽取 1000-2000 篇
  - 由领域专家进行人工标注 (文档类型 + PII实体)
  - 标注结果作为 ground truth
  - 90% 用于评估, 10% 用于蒸馏人工审核
  - 评估集不参与任何训练或类型库构建

更新: 每季度从新文档中抽取补充, 保持评估集与生产分布一致
```

### 10.2 评估指标

| 层级 | 指标 | 目标值 |
|------|------|--------|
| 端到端 | 文档分类准确率 (vs ground truth) | >90% |
| 端到端 | Macro F1 (各类型均衡) | >85% |
| 端到端 | 新型号发现召回率 | >80% (已知新型号中) |
| Tier 0 | PII 实体级 Precision/Recall/F1 | >90% Precision, >85% Recall |
| Tier 1 | 聚类纯度 (簇内主要类型占比) | >80% |
| Tier 1 | 聚类完整性 (同类型文档在同一簇的比例) | >75% |
| Tier 2 | 已知类型匹配精确率 | >90% |
| 蒸馏 | 蒸馏分类器 vs 完整管道一致率 | >92% |

### 10.3 回归测试

```
每次模型更新 (DeBERTa微调/Mistral微调/蒸馏重训练/规则库更新) 后:
  1. 在评估集上跑完整管道
  2. 对比更新前后的各指标
  3. 任一端到端指标下降 > 3% → 阻止上线
  4. 输出 diff 报告: 哪些类型的分类结果发生了变化

频率:
  - 规则库更新: 每次更新后
  - 模型微调: 每次训练完成后
  - 蒸馏重训练: 每次retrain后
  - 全量回归: 每周一次 (即使无变更)
```

## 十一、与Cyera公开架构的关键差异及取舍

| 维度 | Cyera公开架构 | 本方案 | 取舍理由 |
|------|-------------|--------|---------|
| NER模型 | FLAN-T5 (encoder-decoder) | DeBERTa-v3 (encoder-only) | FLAN-T5精度略高(~96% vs ~93% F1), 但延迟高50-100倍。在NER作为中间特征的场景中, 延迟优先 |
| 嵌入模型 | FLAN-T5 Encoder (推断) | BGE-M3 | 专用嵌入模型的对比学习训练目标更适合聚类 |
| 分类融合 | 双路径并行融合 (真值表 + ML) | 四层串行递进 (Tier 0→1→2→3) | 并行融合提供交叉验证, 串行递进更简单。本方案在Tier 3通过独立LLM验证弥补交叉验证的缺失 |
| 聚类方法 | 元数据替换模式匹配 | 结构哈希 + BGE-M3语义精炼 | Cyera方法对半结构化数据高效且O(N), 但对非结构化纯文本文档区分力弱。本方案两阶段覆盖全部数据类型 |
| 串行vs并行 | 专利描述的融合是并行 | 本方案是串行 | 有意取舍: 串行减少了系统复杂度, 但失去了不同路径投票的交叉验证优势。Tier 3质量门部分补偿了这个损失 |
